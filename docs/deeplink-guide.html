미인증 유저는 로그인 후 원래 목적지로 복귀하고, 딥링크 진입 시 뒤로가기는 항상 홈으로 돌아갑니다.

링크 구성 — App Links (외부 웹 → 앱)

목적지URL
매물 상세https://link.estateslug.com/sales/{id}
검색 결과https://link.estateslug.com/search?keyword={검색어}
홈 탭https://link.estateslug.com/home
관심 탭https://link.estateslug.com/favorite
마이 탭https://link.estateslug.com/mypage

링크 구성 — 커스텀 스킴 (estateSlug://)

도메인 없이 동작하는 앱 전용 스킴입니다. 커스텀 스킴은 host 자리가 라우트가 됩니다 (estateSlug://home/의 home). App Links와 동일한 목적지로 매핑됩니다.

목적지URI
매물 상세estateSlug://sales/{id}
검색 결과estateSlug://search?keyword={검색어}
홈 탭estateSlug://home/
관심 탭estateSlug://favorite/
마이 탭estateSlug://mypage/

스킴 표기는 estateSlug·estateslug 모두 동작(매니페스트에 두 표기 선언 + 리졸버 대소문자 무시). assetlinks 없이도 즉시 동작하지만, 브라우저 주소창 직접 입력으로는 열리지 않으며(a 태그·앱·QR 경유는 가능) 미설치 유저에겐 아무 동작도 하지 않는 것이 커스텀 스킴의 한계 — 외부 공유용은 App Links 권장.

링크 구성 — 카카오톡 공유 (KakaoLink)

공유 템플릿의 androidExecutionParams에 route 파라미터를 담는 계약입니다. 카카오톡이 앱을 실행할 때 URI는 kakao{APP_KEY}://kakaolink?route=... 형태가 되고, 라우터가 route 값을 그대로 라우트 토큰으로 해석합니다.

목적지executionParams
매물 상세route=sales/{id}  (현재 ShareKaKao가 발신하는 형태)
검색 결과route=search&keyword={검색어}
탭route=home · route=favorite · route=mypage

앱 미설치 유저는 템플릿의 webUrl(https://link.estateslug.com/...)로 폴백. 송신부는 util/ShareKaKao.kt, 수신 매핑은 DeepLinkResolver의 KAKAO_ROUTE_PARAM 참고.

링크 구성 — FCM 푸시 payload

같은 URL을 data.link에 담아 보냅니다.

{
  "data": {
    "link": "https://link.estateslug.com/sales/12345",
    "body": "관심 매물의 입찰 기일이 3일 남았어요"
  }
}
⚠ data-only로 보내야 합니다. "notification" 키를 함께 넣으면 앱이 백그라운드일 때 시스템 트레이가 알림을 처리해 onMessageReceived가 호출되지 않고 link가 무시됩니다 → 딥링크 라우팅 실패.

링크 규칙

  • host 고정 — link.estateslug.com 외 host의 https 링크는 홈으로 폴백
  • 경로 소문자 — /sales, /search, /home, /favorite, /mypage
  • 검색어 인코딩 — keyword는 URL 인코딩 (공백 → %20)
  • 안전 폴백 — 알 수 없는 경로·형식은 홈으로 (앱이 죽지 않음)
  • 인증 게이팅 — 미로그인 진입 시 로그인 후 원 목적지로 복귀

adb 테스트 명령

커스텀 스킴은 컴포넌트 지정 없이 바로 동작합니다 (에뮬레이터 검증 완료):

# 홈 탭 / 매물 상세 / 검색 — estateSlug·estateslug 표기 모두 동작
adb shell am start -a android.intent.action.VIEW -d "estateSlug://home/"
adb shell am start -a android.intent.action.VIEW -d "estateSlug://sales/16400"
adb shell am start -a android.intent.action.VIEW -d "estateslug://search?keyword=%EC%95%84%ED%8C%8C%ED%8A%B8"

https(App Links)는 assetlinks.json이 아직 없어도 라우터 컴포넌트를 직접 지정하면 확실히 열립니다 (FCM이 타는 경로와 동일).

# 매물 상세 (id=12345)
adb shell am start -n com.estateslug.slug/.deeplink.DeepLinkRouterActivity \
  -a android.intent.action.VIEW \
  -d "https://link.estateslug.com/sales/12345"

# 검색 결과 (keyword=아파트, URL 인코딩)
adb shell am start -n com.estateslug.slug/.deeplink.DeepLinkRouterActivity \
  -a android.intent.action.VIEW \
  -d "https://link.estateslug.com/search?keyword=%EC%95%84%ED%8C%8C%ED%8A%B8"

# 관심 탭
adb shell am start -n com.estateslug.slug/.deeplink.DeepLinkRouterActivity \
  -a android.intent.action.VIEW \
  -d "https://link.estateslug.com/favorite"

assetlinks.json 호스팅 후 실제 App Links 검증까지 확인할 때 (컴포넌트 없이 패키지만 지정 → OS가 intent-filter로 해석):

adb shell am start -a android.intent.action.VIEW \
  -d "https://link.estateslug.com/sales/12345" com.estateslug.slug

# App Links 검증 상태 확인 / 재검증 (Android 12+)
adb shell pm get-app-links com.estateslug.slug
adb shell pm verify-app-links --re-verify com.estateslug.slug
Tip. URL에 ?가 있으면 셸이 글롭으로 해석할 수 있으니 반드시 큰따옴표로 감싸세요. 그래도 안 되면 \?로 이스케이프합니다. 앱을 백그라운드로 내린 뒤 실행하면 백스택(뒤로가기 → 홈) 동작도 검증됩니다.

배포 전 필요 조건 (코드 밖 · 서버/인프라)

  • assetlinks.json 호스팅 — https://link.estateslug.com/.well-known/assetlinks.json에 패키지 com.estateslug.slug + 서명 SHA-256 등록해야 외부 https 링크가 앱으로 자동 오픈(autoVerify). 없으면 브라우저로 감. (FCM 내부 라우팅은 이것 없이도 동작)
  • 서버 푸시 data-only 발송 — 위 payload 형식 준수
  • Firebase 새 앱 등록 — 정식 google-services.json 재발급, Google/Kakao/Naver OAuth 콘솔에 새 패키지 재등록 (패키지 리네임 후속)
  • 디바이스 실측 — 탭 시작점 · 검색 진입 · 백스택 런타임 동작 확인 (빌드는 통과)

후속 (Phase 3 · 기록)
